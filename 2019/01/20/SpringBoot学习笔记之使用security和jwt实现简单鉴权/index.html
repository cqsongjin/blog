SpringBoot学习笔记之使用security和jwt实现简单健全

前言

上一节通过集成mybatis-plus实现了简单增删改查的demo，本节将继续在上一节基础上，为每一个增删改查的操作添加权限控制，普通用户能够执行单挑查询操作以及公共接口的访问，但是对于用户的添加修改操作，则只有管理员有对应的权限，本节将完成这部分功能。

spring-security不经行过多介绍，既然使用了springboot全家桶，在安全控制方面，首选spring-security进行集成。

核心的用户验证功能使用jwt(Json Web Token)技术。jwt的主要执行流程如下：

  1. 客户端发起登陆请求，将用户名和密码信息传输到服务器
  2. 服务器接收客户端的用户名密码，并在后台进行比对，如果认证通过，则将用户信息以及角色权限信息写入到jwt证书，并设置合理的有效期。如果认证为通过，直接返回403状态
  3. 客户端获取到服务端发送的jwt证书，一般情况下为一串加密后的字符串。并在以后的每一次请求中，在请求消息头的Authorization字段中，填入jwt证书字符串
  4. 后台服务器根据配置，拦截需要权限的URL请求，并解析消息中的证书，如果用户证书有效且满足角色或者权限条件，则调用对应请求，如果证书无效或角色权限不满足，则返回403状态码。

补充概念：

JWT和Bearer Token的關係
JWT在使用的時候要在token前面加上字符串「bearer」再填到HTTP請求的header里，這個值就是「Bearer Token」。為什麼要這麼做？它和JWT是什麼關係？

簡單地說，JWT是一種token格式，Bearer Token是一種鑒權方案。

HTTP的header項「Authorization」是在HTTP 1.0引入的，它的值的格式是類型+token，支持多種鑒權方案，bearer只是其中的一種。bearer方案中使用的token是JWT格式，這就是它們之間的關係。

@PreAuthorize注解说明

注解中可以使用el表达式，来声明需要的权限，其中有两个参数说明一下

  • hasAuthority：判断是否具有对应的权限，待比较权限和比较字符串进行全匹配
  • hasRole：判断是否具有对应的角色，如果比较字符串没有ROLE_开头，则加上ROLE_后进行比较

对于GrantedAuthority接口实现类来说是不区分是Role还是Authority，二者区别就是如果是hasAuthority判断，就是判断整个字符串，判断hasRole时，系统自动加上ROLE_到判断的Role字符串上，也就是说hasRole(“CREATE”)和hasAuthority(‘ROLE_CREATE’)是相同的。

总结

本节利用spring-security和jwt实现了简单的权限控制，核心就是创建了一个jwt生成证书和解析证书的工具类，注册了两个过滤器分别用于处理/login请求以及其他请求，同时注册了一个权限服务的provider类，用于验证用户名密码并返回对应的权限，以上组合即可实现一个普通的权限控制。
